# proxmox/configure

Ansible-based automation that configures Proxmox VE hosts after installation:
local storage content types, Terraform API token, Linux bridge creation, and
VLAN-aware bridge configuration.  Runs against the `pve_hosts` group from the
shared Ansible inventory.

## What It Does

### `--build` (or `make build`)

1. Sets up a local Python 3.12 virtual environment (`.venv/`) and installs
   Ansible plus the `community.sops` collection.
2. Regenerates `$_DYNAMIC_DIR/ansible/inventory/hosts.yml` via
   `generate-ansible-inventory`.
3. Runs `playbook.configure-pve.yaml` against the `pve_hosts` inventory group.

The playbook has three plays:

**Play 1 — localhost**: Uses `config_base` role to load the full framework
config and runs `capture-config-fact.yaml` to expose Proxmox config as a fact.

**Play 2 — `maas_region`**: Reads the MaaS region controller's root SSH public key so it
can be synced to Proxmox nodes in Play 3.

**Play 3 — `pve_hosts`**: Configures each PVE node. The `hypervisor.proxmox.configure`
wave runs a pre-check (`proxmox-configure-precheck`) before apply to verify all nodes
are reachable — if any node is unreachable, the wave fails before configuration begins:

| Task file | What it does |
|---|---|
| `sync-maas-ssh-key.yaml` | Adds the MaaS server's root SSH key to PVE's `authorized_keys` so MaaS can reach PVE nodes. |
| `configure-local-storage.yaml` | Sets the content types allowed on local storage pools (e.g. adds `snippets` to `local`). |
| `configure-api-token.yaml` | Creates a Terraform API token (`root@pam!tg-token`) on the PVE node with full privileges and no privilege separation. |
| `configure-bridges.yaml` | Main bridge dispatcher — iterates `pve_bridges` list and calls the appropriate sub-tasks per bridge entry (see below). |

**Bridge configuration (`configure-bridges.yaml`)** processes each entry in `pve_bridges`
in five ordered steps:

1. **Auto-NIC Linux bridges** (`nic == ""`): creates the bridge from the primary NIC
   (`configure-linux-bridge.yaml`). Handles systemd-networkd masking and reboot.
2. **Explicit-NIC Linux bridges** (`nic != ""`): creates a bridge with a named NIC
   (`_configure-explicit-bridge.yaml`). Assumes networkd already masked.
3. **VLAN-aware enablement**: enables `bridge-vlan-aware` on all `technology == "vlan-aware"` bridges
   (`configure-vlan-aware-bridge.yaml`). Applies network config if the kernel state is out of sync.
4. **OVS bridges**: installs OVS packages (automatically, per distro: apt on Debian/Ubuntu,
   EPEL+dnf on Rocky, CRB+EPEL+dnf on RHEL), creates OVS bridge + NIC attachment + optional
   OVSIntPort with host IP (`_configure-ovs-bridge.yaml`). Host IP for OVS is handled inside
   this step via OVSIntPort — Step 5 does not run for OVS bridges.
5. **Host IP assignment** (Linux bridges only): assigns IPs and gateway to non-OVS bridges with
   `host_ip != ""` (`_configure-bridge-host-ip.yaml`). Supports two modes:
   - `host_vlan` set: IP lives on a VLAN subinterface (e.g. `vmbr0.10`), gateway on parent bridge.
   - `host_vlan` absent/null: IP assigned directly to the bridge interface.

### `--test` (or `make test`)

Runs `playbook.configure-pve-test.yaml`, which calls `verify-bridge-config.yaml`
to confirm for each bridge in `pve_bridges`:
- `vlan-aware` bridges: kernel VLAN filtering is active (`vlan_filtering == 1`).
- `ovs` bridges: `ovs-vsctl` lists the bridge.
- Bridges with `host_ip + host_vlan`: the VLAN subinterface exists in the kernel.

Uses the existing inventory; generates it first if not present.

### `--deps` (or `make deps`)

Sets up the Python virtual environment and installs Ansible and
`community.sops` without running any playbooks.

### `--status` (or `make status`)

Prints the path to the current PVE inventory file and its contents.

### `--clean` / `--clean-all`

`--clean` removes the local `tmp/` directory.  `--clean-all` also removes
the `.venv/` directory.

## Inventory

The shared dynamic inventory is generated by `$_GENERATE_INVENTORY --build`.

PVE hosts appear in the `pve_hosts` group via the `role_pve_host` tag in
`pwy-home-lab-pkg.yaml`.  Per-host variables (IP, `pve_bridge_technology`,
`pve_cloud_public_ip`, etc.) come from the `config_params` entry for each
pve-node unit.

## Config

All PVE configuration is read from the framework YAML via `config_base` (Play 1).

| Config key | Purpose |
|---|---|
| `providers.proxmox.config_params.<node>.node_name` | PVE node name (e.g. `pve`, `pve2`) |
| `providers.proxmox.config_params.<node>.bridges` | List of bridge descriptor dicts (see below). If absent, synthesized from legacy flat fields. |
| `providers.proxmox.config_params.<node>.datastore_vm` | VM storage pool name |

**`bridges` list entry schema** (each entry configures one bridge):

| Key | Type | Description |
|---|---|---|
| `name` | str | Bridge interface name (e.g. `vmbr0`, `ovs0`) |
| `nic` | str | Physical NIC to attach. `""` = auto-detect primary NIC (one per host) |
| `technology` | str | `vlan-aware`, `linux`, or `ovs` |
| `host_ip` | str | Host IP to assign to the bridge. `""` = none |
| `host_vlan` | int\|null | VLAN subinterface tag (e.g. `10`). `null` = IP on bridge directly |
| `gateway` | str | Gateway to set. `""` = do not change |
| `comment` | str | Free-text comment in network config |

**Backward compatibility**: if no `bridges` key is present, `generate_ansible_inventory.py`
synthesizes a single-entry `pve_bridges` list from the legacy flat fields
(`bridge_technology`, `vlan_bridge`, `cloud_public_ip`, `gateway`).

## Prerequisites

- `set_env.sh` sourced (provides `_UTILITIES_DIR`, `_DYNAMIC_DIR`, `_CONFIG_DIR`).
- PVE hosts must be running and reachable via SSH (provisioned by the
  `proxmox/install` stage).
- SSH key for the PVE `root` user must be available.
- Python 3.12 must be available on the local machine.
